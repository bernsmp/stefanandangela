## PART 5: OPERATIONAL PROTOCOLS

### Protocol #1: Communication Translation System

**When Stefan Needs to Communicate to Angela:**

Angela needs compressed input. Her processing is: destination → locked doors → questions → done.

**TEMPLATE:**
```
[SITUATION: 1-2 sentences maximum]
[OUTCOME: What we're trying to achieve]
[TOP CONSTRAINT: The one thing that could break this]

Detail available on: [list 2-3 areas]
Which door needs unlocking first?
```

**Example:**
> "We're deciding on quiz software. Need to launch in 30 days. Top constraint: must integrate with existing CRM. Detail available on: pricing tiers, feature comparison, implementation timeline. Which door needs unlocking first?"

**Why This Works:**
- Gives Angela the destination
- Flags the critical constraint
- Offers detail on-demand
- Puts her in driver's seat for prioritization

---

**When Angela Needs to Communicate to Stefan:**

Stefan needs constraint context. His processing is: map system → identify dependencies → optimize.

**TEMPLATE:**
```
Here's where I'm going: [destination in one sentence]
Current blocker: [specific question/decision needed]
Don't need yet: [what context isn't relevant until later]
Ready to execute once you've answered: [specific blocker]
```

**Example:**
> "Here's where I'm going: Launch quiz by end of month. Current blocker: Which software integrates with our CRM? Don't need yet: Long-term roadmap, advanced features, pricing negotiations. Ready to execute once you've answered integration question."

**Why This Works:**
- Gives Stefan the constraint to solve
- Eliminates noise Angela doesn't need
- Creates clear "unlock" moment
- Maintains Angela's momentum

---

**Preference vs. Principle Labeling:**

When stating preferences, EXPLICITLY label:
- "This is my preference: [X]" NOT "This is less effective: [X]"
- "The data shows: [X]" NOT "I think: [X]" (when data-driven)

This prevents Stefan's preferences becoming mandates and preserves Angela's data-driven decision-making.

### Protocol #6: The Stefan-Angela Collaborative Decision Process

**Purpose:** Step-by-step process for making decisions together that leverages your different cognitive strengths at each stage.

**The Protocol:**

2. Angela reads and confirms: "Here's the 2-sentence version - is this right?"
3. Both agree on simplified problem definition before moving forward
4. Document: "The problem we're solving is [X], not [Y]"

**Red Flag:** If you can't agree on problem definition, you'll waste weeks on wrong solutions.

#### STAGE 2: INFORMATION GATHERING

**This is where friction happens most. The solution is PARALLEL WORK.**

**Stefan's Contribution:**

**Angela's Contribution:**

**Time Allocation:**
- Stefan: Up to 2 weeks for systems analysis (honors his need for depth)

- **No judgment of the other's approach during this phase**

**Why This Works:**  
Eliminates the "Stefan needs more time" vs "Angela wants to move" friction. Both approaches operate simultaneously. Angela's speed is an advantage, not a pressure on Stefan to rush. Stefan's depth is thoroughness, not a blocker to Angela's momentum.

**The Protocol:**

3. Angela presents: 2-3 pattern-matched approaches from experience

5. Generate 1-2 hybrid alternatives that leverage both approaches

**Target:** 5-7 viable alternatives total (Stefan's models + Angela's patterns + hybrids)

#### STAGE 4: DECISION SELECTION

**Stefan's Contribution:**
- Pressure-tests logic and reasoning
- Models potential outcomes and risks
- Validates that solution accounts for system constraints

**Angela's Contribution:**
- Reality-checks execution feasibility
- Validates that solution is implementable
- Confirms pattern matches real-world experience

**FOR TYPE 1 DECISIONS (Irreversible, >$50K, >6 months):**
1. Stefan validates: Logic is sound, constraints are mapped, optimization is reasonable
2. Angela validates: Execution path is feasible, team has capacity, timeline is realistic
3. **BOTH must agree** OR go to tiebreaker:
   - If Stefan sees systemic risk → His veto (prevents catastrophic mistakes)
   - If Angela sees execution impossibility → Her veto (prevents unrealistic plans)
4. Document: Decision + rationale + what would change our mind

**FOR TYPE 2 DECISIONS (Reversible, <$50K, <30 days):**
1. Quick assessment using Stefan's framework
2. **Default to Angela's speed** (reversible means we can iterate)
3. Stefan has right to flag: "This looks reversible but it's actually not because..."
4. Document: Decision + when we'll review results

**Disagreement Protocol - 3 Levels:**

**LEVEL 1: Standard Disagreement** (Use Stage 4 protocol above)
- Stefan validates structure; Angela validates momentum
- Apply Decision Matrix category (Type 1 or Type 2)
- Use veto rules if applicable
- Expected resolution time: Same meeting

**LEVEL 2: Cognitive Mismatch** (Use 48-hour table)
- Triggered when: Different information needs create impasse
- Name the specific pattern causing tension (e.g., "Stefan's need for constraint map vs. Angela's need for movement")
- Each gathers data supporting their view
- Reconvene with evidence in 48 hours
- Expected resolution time: 48 hours

**LEVEL 3: True Deadlock** (After 48 hours, still no agreement)

Triggered when:
- Both still disagree AND
- Decision can't be delayed AND
- Neither will yield

**Three Options:**

**Option A: CEO Veto**
- Stefan decides (his company, he bears consequences)
- Angela executes fully committed even if she disagrees
- Set explicit review date (30-90 days) to assess outcome
- Use this when: Time pressure is extreme OR systemic risk is high

**Option B: Test Both**
- If truly reversible, run A/B test of both approaches
- Define success metrics before testing
- Time-bound experiment (30-60 days)
- Winner determined by data, not opinion
- Use this when: Both approaches are testable AND cost of testing < cost of wrong choice

**Option C: External Tiebreaker**
- Bring in board member/advisor for outside perspective
- Both present case in 10 minutes each
- Advisor questions both, makes recommendation
- Stefan retains final decision but with informed input
- Use this when: Strategic significance is high OR neither option A nor B feels right

**NEVER use Level 3 for:** Reversible decisions (just pick one and iterate), decisions that can wait, or ego-driven disagreements

**Frequency Check:** If you hit Level 3 more than once per quarter, your problem isn't deadlock - it's insufficient protocol design. Revisit Parts 2-5.

## PART 8.5: CRISIS DEFINITION & RESPONSE PROTOCOL

### What Qualifies as a "Crisis"

**A situation qualifies as "crisis" if it meets 2 or more of these criteria:**

| Criterion | Threshold |
|---|---|
| **Revenue Impact** | >$50K at risk in next 7 days |
| **Customer Impact** | >500 customers affected or at risk |
| **Reputational Impact** | Media/social attention likely or already occurring |
| **Operational Failure** | Core system down >2 hours OR critical process blocked |
| **Time Sensitivity** | Must decide/act in <24 hours with significant consequences |
| **Legal/Compliance** | Regulatory violation or legal exposure |

---

### Crisis Response Protocol

**STEP 1: Crisis Declaration**

**Angela declares crisis** (uses Impact-First Diagnostic Pattern 13 automatically)
- States: "I'm calling this a crisis. Here's the impact calculation: [X]"
- Identifies which criteria are met (minimum 2)
- Proposes immediate action

**STEP 2: Stefan Confirmation** (5-minute maximum)

Stefan can either:
- **Confirm:** "Agreed, it's a crisis. You lead, I'll support."
- **Downgrade:** "This meets only 1 criterion, not 2. Here's why: [specific reason]. Let's use normal protocol."
- **Escalate:** "It's worse than you think. Here are additional systemic risks: [X]"

**STEP 3: If Both Agree It's Crisis**

**Angela takes command** (Protocol #2, Type 2C from Decision Matrix)
- Makes all tactical decisions
- Assigns tasks without consensus-building
- Operates in "clear, fast, decisive" mode

**Stefan feeds critical context only** (2-sentence updates)
- "Here's the system constraint you need to know: [X]"
- "This action will cause [Y] second-order effect"
- Does NOT slow her down with full analysis

**Normal protocols suspended** until crisis resolved

**STEP 4: If Stefan Disagrees It's a Crisis**

Both spend 5 minutes on impact calculation:
- Stefan: "Here's why I don't think this meets crisis threshold: [specific analysis]"
- Angela: "Here's why I do: [quantified impact]"
- Apply the 2+ criteria test objectively
- If meets 2+ criteria → It's a crisis, Angela leads
- If doesn't meet criteria → Use normal decision protocols

---

### Crisis Ends When

**Automatic end triggers:**
- Immediate threat neutralized (revenue secured, customers protected, system restored)
- OR 48 hours passed (automatically shift to normal protocols even if not fully resolved)

**Why 48-hour limit:**  
Extended "crisis mode" creates team burnout and prevents proper systems thinking. After 48 hours, shift to normal protocols even if problem persists.

---

### Post-Crisis Protocol (Within 7 Days)

**30-Minute Debrief:**
1. What triggered the crisis?
2. Did our response work? What would we change?
3. Could Stefan's systems thinking have prevented this?
4. Could Angela's speed have prevented worse outcomes?
5. What protocol adjustment prevents recurrence?

**Document:** Crisis, response, outcome, lesson

**Update:** Relevant protocols based on what you learned
